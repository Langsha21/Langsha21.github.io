介绍

PE文件是Windows平台的通用可执行文件格式,是微软在基于UNIX的COFF(通用对象文件格式)的基础上制作而成,虽然最初设计用于提高可移植性,但事实上只可在Windows平台上运行

PE文件格式

种类 主扩展名
可执行系列 EXE,SCR
驱动文件系列 SYS,VXD
库系列 DLL,OCX,CPL,DRV
对象文件系列 OBJ

OBJ文件虽然规定为PE文件,但是无法以任何形式执行,故而基本不用关注,其他的文件都可以通过调试器,shell等不同的方式来运行.

基本结构

这次我们以记事本(notepad.exe)作为例子,使用十六进制编辑器打开 上图为记事本文件的头部,其具有普通的PE文件的格式

DOS头到节区头为PE头,而下面的节区合称为PE体

文件中使用偏移,在内存中使用虚拟地址来表示位置.

文件的内容一般可分为代码(.text),数据(.data)和资源(.rsrc)节分别存储.

PE的头和各个节区的尾部都存在一个区域,使用了00填充,被称为NULL填充

VA&RVA

VA(虚拟地址)为进程虚拟空间中的绝对地址,而RVA(Relative Virtual Address,相对位移地址)指从某个基址开始的相对地址,VA和RVA的换算为: \[ RVA+ImageBase=VA \]

PE头内的信息大多数以RVA形式存在,而非VA.原因是PE文件(主要是DLL)加载到进程虚拟地址的特定位置的时候,这个位置可能已经加载了其他的PE文件(DLL),那么此时必须使用重定向(Relocation)来将其加载到其他的空白位置中.如果此时仍旧使用VA则无法正常访问.

PE头

PE头由许多的结构体组成

DOS头 IMAGE_DOS_HEADER

这个结构体的大小为64字节 其中有两个重要成员: 1. e_magic: DOS签名(固定为MZ,十六进制为4D5A) 2. e_lfanew: 指示NT头的偏移(NT头名称为IMAGE_NT_HEADERS)

Fun:MZ的来源为设计了DOS文件格式的Mark Zbikowski

在图中,e_magic为MZ,而e_lfanew为00000100

DOS存根 stub

DOS存根(stub)是一个可选段,在DOS头下方,大小不固定.

DOS存根由代码和数据混合而成 上图中40-4D的区域为16位的汇编代码,32位或者64位的Windows系统不会执行这段代码(因为将该文件识别为PE文件,故而会忽略这段代码),在DOS环境下或者是使用DOS调试器(debug.exe)调试该文件可以执行这段代码

文件头结构体

我们先关注下IMAGE_OPTIONAL_HEADER32中的几个重要成员,他们的值若设置错误程序将无法运行

1. Magic

在IMAGE_OPTIONAL_HEADER32中Magic值为10B,在IMAGE_OPTIONAL_HEADER64中为20B

2. AddressOfEntryPoint

规定为持有EP(Entry Point,入口处)的RVA值,也即程序最先执行的代码起始位置

3. ImageBase

优先装入地址,执行文件时,系统会首先尝试将文件载入这个地址,如果这个地址被占用,则再考虑载入其他地址

4. SectionAlignment, FileAlignment

PE的Body部分被划分为若干节区,这些节存储着不同类型的数据,FileAlignment制定了节区在磁盘中的文件中的最小单位,SectionAlignment则指定了文件在内存中的最小单位(这两个值可能相同也可能不同),磁盘文件中或者内存文件中的节区大小必定为SectionAlignment或者FileAlignment值的整数倍

5. SizeOfImage

指定PE Image(PE文件被加载到内存中被称为PE Image)在虚拟内存所占空间大小.一般而言,文件的大小和加载到内存中的大小是不一致的

6. SizeOfHeaders

用于指出整个PE头的大小,该值也必须是FileAlignment值的整数倍

7. Subsystem

用于区分系统驱动文件(*.sys)和普通的可执行文件(*.exe, *.dll) 其可用的值为: | 值 | 含义 | 备注 | | --- | --- | --- | | 1 | Driver文件 | 系统驱动(例如: ntfs.sys) | | 2 | GUI文件 | 窗口应用程序(例如: notepad.exe) | | 3 | CUI文件 | 控制台应用程序(例如: cmd.exe) |

8. NumberOfRvaAndSizes

用于指定DataDirectory(可选头结构体中的最后一个成员)数组的个数,虽然在定义中规定了数组的大小为16,但是实际中并不一定是16,PE装载器读取NumberOfRvaAndSizes确定DataDirectory的值

9. DataDirectory

由DataDirectory结构体组成的数组,数组的每一项都有预定的值

索引 名称 描述
0 Export Table 导出表，列出该模块导出的所有符号（函数和变量）。
1 Import Table 导入表，列出该模块导入的所有符号（函数和变量）。
2 Resource Table 资源表，包含该模块的所有资源（如图标、字符串、对话框等）。
3 Exception Table 异常表，提供与该模块相关的异常处理信息。
4 Certificate Table 证书表，包含数字签名证书，用于验证模块的完整性和来源。
5 Base Relocation Table 基址重定位表，当模块不能加载到其首选基地址时，用于修正地址。
6 Debug Data 调试数据，包含调试信息。
7 Architecture Data 特定架构数据（通常未使用）。
8 Global Pointer Table (GPT) 全局指针表（未使用，在PE文件中通常为空）。
9 Thread Local Storage (TLS) 线程本地存储表，包含线程局部存储的初始化数据。
10 Load Configuration Table 加载配置表，包含系统加载模块时需要的配置信息。
11 Bound Import Table 绑定导入表，加速模块加载的绑定导入信息。
12 Import Address Table (IAT) 导入地址表，包含所有导入函数的地址。
13 Delay Import Descriptor 延迟导入描述符，包含延迟加载的导入模块信息。
14 COM Runtime Descriptor COM运行时描述符（.NET头），包含该模块的COM（组件对象模型）相关信息，主要用于.NET模块。
15 Reserved 保留，未使用。

其中较为重要的是IMPORT/EXPORT和TLS.

RVA to RAW

现在我们要去钻研下PE文件从磁盘映射到内存的内容,PE文件从磁盘映射到内存的同时,每个节区都要完成内存地址与文件偏移之间的映射,这被称为RVA to RAW,其方法为:

  1. 查找RVA所在节区
  2. 使用简单的公式计算文件偏移(RAW)

根据IMAGE_SECTION_HEADER的结构体,其换算公式为: \[ RAW-PointerToRawData=RVA-VirtualAddress \\ RAW=RVA-VirtualAddress+PointerToRawData \]

需要注意,这其中不涉及到ImageBase,这其中的VirtualAddress事实上是节区的RVA

IAT

IAT(Import Address Table)可能是学习PE头中最难的一关了.IAT中保存的内容与Windows操作系统的核心进程,内存,DLL结构有关.IAT的本质就是一种表,记录程序此时正在使用库中的哪些函数.

DLL

在写到IAT之前先写下DLL(Dynamic Link Library,动态链接库)的知识.其支撑起了整个Windows OS.

在16位的DOS中不存在DLL这一概念,而只有库的概念.如同在C语言中执行include操作一般,要调用一个库函数,必须先将这个库include,也即包含这个库的代码,但是如果延续这种做法,将会使得程序相当的臃肿,浪费磁盘以及内存空间.

于是DLL的概念相应而出,其含义为: - 不要将库包含在程序当中,而是将其单独组成DLL文件,需要时调用即可. - 内存映射技术可以让加载后的DLL代码,资源在多个程序间共享. - 更新库的时候只需要替换相关的DLL文件即可.

加载DLL有两种方法:显式链接(Explicit Linking)和隐式链接(Implicit Linking).前者在程序使用DLL时加载,使用完毕后释放资源.后者在程序开始的时候就加载DLL,程序结束时才释放资源.

IAT提供的机制就与DLL的隐式链接有关.现在我们使用IDA调试下notepad.exe文件,并且手动F7到调用kernel32.dll的时候

此处的程序运行到了调用函数的代码,注意到此时使用了CALL,而cs:__imp_GetModuleHandleW其实并没有函数(汇编代码),cs:__imp_GetModuleHandleW本身是位于idata(也即IAT表数据)位置,而运行程序的时候,PE装载器将会把需要调用的函数的地址写到位于idata中的cs:__imp_GetModuleHandleW(所有的API调用都采用这种方式)

为什么不直接写上你所需要的函数的地址呢?很简单,因为你无法写入你所不知道的东西,在不同的机器,不同版本的Windows中,DLL的版本不同,函数的地址也不同,故而如果写死在内存中(DOS时代即是如此),要么调用错误函数,要么程序错误.

还有一个原因是因为重定位,如果你预先设定了函数地址,而在执行过程中这个地址被其他的DLL所占用直到函数调用,那么就会出现问题.

4. FirstThunk - IAT (Import Address Table)

最初的两个字节值为Ordinal,为库中函数固有的编号,后边为函数名称的字符串

这个时候你会发现INT表和IAT表此时指向的RVA地址都是0x252B8,不过有时候这两者指向的也不尽然全是相同的(要不然就没有使用两种表的必要了)

Notice:Windows的DLL带有自身固定的ImageBase,并不会出现需要DLL重定位的情况.

EAT

EAT使得不同的应用程序可以调用库文件中提供的函数,通过EAT我们能求得相应库文件中所需函数的起始地址.PE中的EAT保存着导出信息.并且PE文件中有且仅有一个用于说明库EAT的IMAGE_EXPORT_DIRECTORY结构体

Notice:EAT是对于库而言的,而相对应的IAT是对应于PE文件的.

IMAGE_EXPORT_DIRECTORY

下面列举其中重要的成员: | 项目 | 含义 | | --- | --- | | NumberOfFunctions | 实际Export函数的个数 | | NumberOfNames | Export函数中具名的函数个数 | | AddressOfFunctions | Export函数地址数组 (数字元素个数=NumberOfFunctions) | AddressOfNames | 函数名称地址数组 (数字元素个数=NumberOfFunctions) | | AddressOfNameOrdinals | Ordinal地址数组 (数字元素个数=NumberOfFunctions) |

从库中获取函数地址的API为GetProcAddress()函数,该API引用了EAT来获取指定的API的地址.下面写下其如何获取函数地址.理解了这点就相当于理解了EAT.

GetProcAddress()操作原理
  1. 利用AdrressOfNames成员转到"函数名称数组"
  2. "函数名称数组"中的成员存储着函数字符串地址,通过比较(使用strcmp函数)字符串,查找指定的函数名称.(此时数组的索引称为name_index).
  3. 利用AddressOfNameOrdinals成员转到Ordinal数组.
  4. 在Ordinal数组中查找Ordinal[name_index]的值
  5. 利用AddressOfFunctions成员转到"函数地址数组"(EAT)
  6. 在"函数地址数组"中将刚刚求得的ordinal作为数组的索引,获得指定函数的起始地址,也即获取EAT[Ordinal[name_index]]的值

4.函数地址

最后我们来查询下ActivateActCtxWorker函数的地址.这个时候我们要查询函数地址数组,跳转到AddressOfFunctions的RWA地址.

图中蓝色的区域代表函数地址数组,与上述一致,我们计算出Ordinal值后,Ordinal值作为函数地址数组中的索引.在这个例子里面ActivateActCtxWorker函数的Ordinal值为6,那么函数的地址就是AddressOfFunctions[Ordinal]的值,在表中这个值为0x000206A0.
